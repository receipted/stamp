## Deep Technical Work

### Sieve Verification Architecture (Opus-level session)

**Seven seams where a stochastic actor can spoof:**
- A: Input capture (serialization controls the hash)
- B: Function source pinning (import hooks, .pth files, monkey-patching)
- C: Execution environment (compromised libpython, LD_PRELOAD, modified hashlib.so)
- D: Output serialization (json.dumps without sort_keys, custom handlers)
- E: Receipt construction (wrong variables, broken chain)
- F: Receipt storage (filesystem hooks, FUSE, symlinks)
- G: Verification (verifier code written by same actor = compromised)

**Structural solution:**
- Harness must be ~18 lines — too simple to hide spoofing
- Human reads it once, pins the hash with `sha256sum harness.py`
- Human runs it directly (not through Shadow)
- Integrity ≠ Correctness (receipt proves what ran, not that it was right)

**Core value proposition (honest version):**
> The substrate provides tamper-evident, independently verifiable execution integrity for semantic transformations, with an explicit and minimal trust boundary that the user controls.

**What it CAN prove:** execution integrity, temporal ordering, transformation provenance, tamper evidence, cross-instance convergence
**What it CANNOT prove:** correctness of the sieve function, authenticity of original input, absence of hardware compromise, that the human understood the policy they ratified
**Irreducible trust:** the human trusts the OS + Python runtime, reads 18 lines once, ratifies three hashes once
